# Spec 駆動開発ワークフロー

このプロジェクトでは、実装前に Spec（要件・設計・受け入れ条件）を定義し、合意してから開発する。

## 基本方針
- 実装は Spec に基づいて行う。
- Spec が未作成または未合意の場合、コード変更を開始しない。
- 変更要求ごとに Spec を 1 ファイル作成する。
- Spec は日本語で記載する。

## 標準フロー
1. 要求の整理
- 背景、目的、スコープ、非スコープを明確化する。

2. Spec 作成
- `docs/spec_template.md` を使って Spec を作成する。
- 配置先は `docs/specs/`、ファイル名は `YYYYMMDD_snake_case.md` で固定する。
- 例: `20260215_git_pr_cycle.md`

3. レビュー・合意
- 受け入れ条件（Acceptance Criteria）を中心にレビューする。
- 合意後に実装着手する。

4. 実装
- Spec のタスク分解に従って実装する。
- 変更時は Spec の関連箇所を更新する。

5. 検証
- Spec の受け入れ条件に対してテスト結果を記録する。

6. PR
- PR 本文に対象 Spec のパスを記載する。
- 例: `Spec: docs/specs/20260215_git_pr_cycle.md`

## 運用ルール
- 1 PR = 1 主要 Spec を原則とする。
- 仕様変更が発生した場合は、実装より先に Spec を更新する。
- 未確定事項は Spec の「未確定事項」に残し、判断ログを追記する。
- コミットメッセージは `spec:<spec_id> <内容>` 形式とする。
  - 例: `spec:20260215_git_pr_cycle PRテンプレートを追加`
- PR タイトルは `[<spec_id>] <要約>` 形式とする。
  - 例: `[20260215_git_pr_cycle] Spec駆動運用を追加`
